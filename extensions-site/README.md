# extensions-site

A SQLite-driven landing site for the sqlite-wasm extension registry,
served by `sqlite-wasm-httpd`. Lists 220+ shipped extensions with a
client-side search bar and a per-extension detail page; also exposes
the catalog as a JSON API.

```
   /                       landing page with search
   /ext/<name>             per-extension detail
   /api/extensions.json    full registry as JSON
   /api/ext/<name>.json    per-extension as JSON
   /health                 ok
```

Everything is served from a single SQLite file (`registry.db`)
generated by `build.py` from the canonical sources:
- `provenance/extensions.db`  shipped catalog
- `registry/index.json`        rendered registry entries (exports, checksums, sizes)
- `registry/candidates.json`   wishlist + status decisions

Route handlers live in `registry.db`'s `routes` table; each
handler is a SQL query that materializes HTML or JSON. No wasm
handlers, no external templates at request time  the whole site
is "look up data in the same DB and return the response."

## Local development

```bash
# 1. Refresh the source data (only needed when the catalog has changed)
python3 provenance/scan.py
python3 provenance/build_registry.py

# 2. Build the site DB
python3 extensions-site/build.py
# -> wrote extensions-site/registry.db  226 extensions, 421888 bytes

# 3. Serve locally on http://localhost:8080
cargo build --release -p sqlite-wasm-httpd
./target/release/sqlite-wasm-httpd \
    --db extensions-site/registry.db \
    --port 8080
```

The browse + search experience uses the embedded JSON in the
landing page (no XHR), so it works offline.

## Container build (local)

```bash
docker build -t extensions-site -f extensions-site/Dockerfile .
docker run --rm -p 8080:8080 -e PORT=8080 extensions-site
```

The image is two stages:
1. **builder**  rust:1.83-bookworm; compiles `sqlite-wasm-httpd`
2. **db-builder**  python:3.12-slim; runs `build.py` to generate
   `registry.db` from the catalog data
3. **runtime**  debian:bookworm-slim; just the binary + the DB.
   Total image size ~80 MB (~50 MB debian base + ~30 MB binary +
   ~412 KB DB).

## GCP / Cloud Run deployment

Cloud Run is the natural fit  stateless container, scale to zero,
free HTTPS, $PORT convention.

### One-time setup

```bash
PROJECT_ID=your-project
REGION=us-central1

# Artifact Registry repo (one-time)
gcloud artifacts repositories create extensions \
    --repository-format=docker \
    --location=$REGION \
    --description="sqlite-wasm registry images"

# Cloud Build needs Artifact Registry + Cloud Run deploy permissions:
PROJECT_NUMBER=$(gcloud projects describe $PROJECT_ID --format='value(projectNumber)')
gcloud projects add-iam-policy-binding $PROJECT_ID \
    --member=serviceAccount:${PROJECT_NUMBER}@cloudbuild.gserviceaccount.com \
    --role=roles/run.admin
gcloud projects add-iam-policy-binding $PROJECT_ID \
    --member=serviceAccount:${PROJECT_NUMBER}@cloudbuild.gserviceaccount.com \
    --role=roles/iam.serviceAccountUser
gcloud projects add-iam-policy-binding $PROJECT_ID \
    --member=serviceAccount:${PROJECT_NUMBER}@cloudbuild.gserviceaccount.com \
    --role=roles/artifactregistry.writer
```

### Deploy

```bash
gcloud builds submit \
    --config=extensions-site/cloudbuild.yaml \
    --substitutions=_REGION=$REGION,_SERVICE=extensions-site \
    .
```

This builds the container, pushes both `:$SHORT_SHA` and `:latest`
tags to Artifact Registry, and deploys to Cloud Run with:
- `--allow-unauthenticated`  public site
- 256 MiB memory, 1 vCPU
- scale to zero (`--min-instances=0`)
- max 10 instances
- 80 concurrent requests per instance
- 30s request timeout

Cloud Run prints the service URL at the end of the deploy
something like `https://extensions-site-xxxxxxxx-uc.a.run.app`.

### Updating the catalog

Re-run `provenance/scan.py` + `provenance/build_registry.py` against
the latest extensions/ tree (this is what `make ext` already does),
then re-submit the same `gcloud builds submit` command. Cloud Build
rebuilds the image with the fresh catalog and rolls Cloud Run to
the new revision.

## Architecture: why SQL handlers?

The `routes` table is consulted on every request. Each route's
handler is a SQL query bound with `:method`, `:path`, `:query`,
`:body`, `:remote` (and `:headers` after PR #b29). The query's
result row carries `body` / `status` / `ctype` columns the
dispatcher uses to build the HTTP response.

For this site that means:
- The landing page is one SQL row with the entire HTML body as a
  constant string. Build time bakes the data in.
- The per-extension page is a CASE expression: NULL extension
  yields a 404 page; matched extension yields the templated
  detail page via `replace(...)` against a static skeleton.
- The JSON APIs use `json_object()` + `group_concat()` directly.

No SQL injection: every dynamic value flows through SQLite's
own `replace()` or `json_object()`  it's already escaped for the
output context. The `:path` parameter is a bound parameter, not
string-concatenated.

## Artifact distribution: in-database CAS

The `.component.wasm` binaries live alongside the metadata in the
same `registry.db`  a content-addressed `assets` table baked into
the image at build time. The httpd's `blob` route kind serves
them directly with no Value-type roundtrip:

    GET /asset/<name>        -> raw .component.wasm bytes
    GET /asset-info/<name>   -> { name, size_bytes, sha256, content_type, version, url }

The whole catalog at the time of this writing is **191 artifacts,
~88 MB binary**  small enough that the SQLite file fits
comfortably inside a single container layer (final image ~186 MB
including the debian base + httpd binary). One image rebuild
ships a versioned catalog snapshot; no separate object store, no
egress fees, no out-of-band DNS or sync to coordinate.

### Schema

```sql
CREATE TABLE assets (
    name         TEXT PRIMARY KEY,
    blob         BLOB NOT NULL,
    size_bytes   INTEGER NOT NULL,
    sha256       TEXT NOT NULL,
    content_type TEXT NOT NULL DEFAULT 'application/wasm',
    version      TEXT
);
```

Populated by `extensions-site/build.py` walking
`extensions/*/target/wasm32-wasip2/release/*_extension.component.wasm`
and inserting one row per artifact. Pre-requisite: every
extension built locally before `extensions-site/build.py` (or
before `docker build`). `make ext NAME=<name>` produces one
artifact; loop over the catalog in CI to build the full set.

### How the `blob` route kind works

`sqlite-wasm-httpd`'s router supports four kinds (sql, static,
wasm, blob). The blob kind takes a SQL handler that returns ONE
blob value (first column of first row) and streams it back as
the response body without going through the Value-type conversion
that hex-encodes BLOBs in the `sql` kind. The handler text for
the asset route:

```sql
SELECT blob FROM assets
 WHERE name = substr(:path, length('/asset/') + 1)
```

Response headers:
- `Content-Type: application/wasm` (from the row's `ctype` column;
  could be derived from `assets.content_type` if we want different
  types per file)
- `Cache-Control: public, max-age=31536000, immutable`  the URL
  is identity-keyed; rebuilding the catalog re-deploys to a fresh
  Cloud Run revision with a fresh URL identity at the same path
- `Access-Control-Allow-Origin: *`

Verified end-to-end: the SHA-256 of `curl /asset/jwt` matches
`SELECT sha256 FROM assets WHERE name='jwt'` byte-exact (CAS
integrity check).

### Updating the catalog

Re-run `provenance/scan.py` + `provenance/build_registry.py` +
`extensions-site/build.py`, then `gcloud builds submit`. The image
rebuild bakes the fresh assets table; Cloud Run rolls forward
with zero downtime (new revision serves traffic while the old
finishes existing requests).

### Cost model

The whole thing runs entirely on GCP:
- **Cloud Run** charges per request-second after the free tier
  (2M requests/month free, ~180k vCPU-seconds free, ~360k GiB-seconds
  free). Realistic registry traffic stays in the free tier.
- **Artifact Registry** charges ~$0.10/GB-month for the image.
  Image is ~186 MB so ~$0.018/month.
- **No egress charges** for traffic into GCP, and Cloud Run egress
  is bundled into the same pricing  no separate bill for serving
  88 MB downloads.

Effective cost: pennies per month at modest traffic; only scales
with the request count, not data volume.
