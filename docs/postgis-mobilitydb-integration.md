# Deploying postgis + mobilitydb into sqlink — handoff guide

End-to-end deployment guide for shipping the **postgis** and
**mobilitydb** DataFission wasm shims as loadable SQLite extensions
into sqlink.

Scope: build → compose → bridge → load → smoke. This document does
**not** cover DataFission internals or shim development.

A sibling guide for the DuckDB path
([`ducklink`](https://github.com/tegmentum/ducklink)) lives at
`~/git/ducklink/docs/postgis-mobilitydb-integration.md` and shares
~80% of the surface area — the compose step and bridge regeneration
are identical; only the loadable-extension format and load command
differ.

---

## 1. Architecture overview

Two artifacts ship per shim:

```
postgis-wasm           ─┐                              ┌─ libpostgis_sqlite_bridge.{dylib,so,dll}
  + 14 dep components   │  wac plug                    │   (loaded into sqlink at runtime)
  + proj-composed       ├────────────► postgis-       ┤
  + geos-wasm           │              composed.wasm   │
  + …                  ─┘              (21 MB)
```

- **`<shim>-composed.wasm`** — the upstream wasm shim component
  (`postgis-wasm` or `mobilitydb-wasm`) pre-composed with every dep
  it imports (proj, geos, mdb-temporal, etc.). Built once, hashed,
  pinned.
- **SQLite bridge crate** (`{shim}-sqlite-bridge`) — a thin native
  loadable extension generated by `sqlink-shim-codegen` from a
  shim-interface SQLite DB. Uses rusqlite's
  `create_scalar_function` / `create_aggregate_function` / `VTab`
  traits to surface every shim function as a native SQLite call.
  Embeds `wasmtime`, loads the composed shim at extension init time,
  and dispatches one row at a time (SQLite is not vectorised).

The bridge is generated; the composed wasm is target-agnostic.
sqlink picks the bridge up from a directory on disk; the composed
wasm is found via a `<SHIM>_SHIM_WASM` env var the bridge reads at
init.

### Where things live in the dev tree

| Path | What |
|---|---|
| `~/git/postgis-wasm/` | postgis shim source + `scripts/compose.sh` |
| `~/git/mobilitydb-wasm/` | mobilitydb shim source |
| `~/git/datafission/extensions/postgis/deps/postgis-composed.wasm` | pinned composed artifact for postgis |
| `~/git/datafission/extensions/postgis/deps/postgis-composed-pin.txt` | upstream commit + blake3 |
| `~/git/datafission/extensions/mobilitydb/deps/mdb-temporal-wasm.wasm` | mobilitydb temporal component |
| `~/git/datafission/extensions/mobilitydb/deps/postgis-composed.wasm` | postgis dep for mobilitydb |
| `~/git/sqlink-shim-codegen/` | bridge generator targeting SQLite |
| `~/git/postgis-sqlite-bridge/` | generated bridge crate (postgis) |
| `~/git/mobilitydb-sqlite-bridge/` | generated bridge crate (mobilitydb) |
| `~/git/shim-bridge-smoke-tests/` | end-to-end smoke runner |
| `~/git/shim-sql-preprocess/` | CLI that rewrites operator/cast syntax before send |

sqlink itself does not need a code change for this integration —
it consumes the bridge artifact via its existing extension-loading
path. The bridge is just another loadable extension as far as the
host is concerned.

## 2. Prerequisites

### Build environment

- **Rust** stable (1.75+); `wasm32-wasip2` target installed:
  `rustup target add wasm32-wasip2`
- **wasi-sdk-33** at `~/git/postgis-wasm/external/wasi-sdk-33.0-<platform>`
  (only needed when rebuilding the composed shim — bridge build
  does not need it)
- **`wac`** for component composition:
  `cargo install wac-cli`
- **`wasm-tools`** for component inspection:
  `cargo install wasm-tools`
- **SQLite CLI with extension support**. macOS `/usr/bin/sqlite3`
  is compiled `-DSQLITE_OMIT_LOAD_EXTENSION` and refuses `.load`.
  Use homebrew sqlite (`/opt/homebrew/bin/sqlite3`) or any distro
  sqlite that ships extension support.

### Repository checkouts

The bridges path-depend on three datafission workspace crates,
so all of these must sit side-by-side:

```
~/git/
├── datafission/                  # workspace root
├── postgis-wasm/                 # upstream shim
├── mobilitydb-wasm/              # upstream shim
├── postgis-sqlite-bridge/        # generated bridge
├── mobilitydb-sqlite-bridge/
├── sqlink-shim-codegen/
├── shim-sql-preprocess/
├── shim-bridge-smoke-tests/
└── sqlink/                       # this repo
```

## 3. Build flow

### 3.1 Compose the shim wasm

The composed shim is checked in under
`extensions/<shim>/deps/<shim>-composed.wasm` and changes only when
upstream changes. For an existing checkout, **skip this step**
unless you've bumped the pin.

To rebuild against the pinned upstream:

```sh
cd ~/git/datafission
scripts/rebuild-postgis-composed.sh           # postgis
extensions/mobilitydb/scripts/bump-pin.sh     # mobilitydb
```

Both verify the resulting blake3 matches the pin file. A mismatch
means either the upstream commit moved or a dep changed.

### 3.2 Generate the bridges

```sh
cd ~/git/sqlink-shim-codegen
cargo run --release -- \
    --interface /tmp/postgis-interface.sqlite \
    --out ~/git/postgis-sqlite-bridge

cargo run --release -- \
    --interface /tmp/mobilitydb-interface.sqlite \
    --out ~/git/mobilitydb-sqlite-bridge
```

The `--interface` SQLite file is the shim's published surface
(scalars, aggregates, types, casts, etc.). The codegen pipes every
emitted `.rs` through `rustfmt --edition 2021`, so the bridge
crates stay fmt-clean by construction.

### 3.3 Build each bridge

```sh
(cd ~/git/postgis-sqlite-bridge && cargo build --release)
(cd ~/git/mobilitydb-sqlite-bridge && cargo build --release)
```

The build produces `target/release/lib<bridge>.dylib` (macOS),
`.so` (Linux), or `.dll` (Windows). SQLite loads the unmodified
shared library directly — no metadata-footer wrap step.

## 4. Integrating into sqlink

sqlink consumes extensions from a directory tree under
`~/git/sqlink/extensions/` (~230 components already shipped). The
shim bridges slot in as additional entries.

### 4.1 Drop the bridge artifacts in

```sh
mkdir -p ~/git/sqlink/extensions/postgis-bridge
cp ~/git/postgis-sqlite-bridge/target/release/libpostgis_sqlite_bridge.dylib \
   ~/git/sqlink/extensions/postgis-bridge/

mkdir -p ~/git/sqlink/extensions/mobilitydb-bridge
cp ~/git/mobilitydb-sqlite-bridge/target/release/libmobilitydb_sqlite_bridge.dylib \
   ~/git/sqlink/extensions/mobilitydb-bridge/
```

### 4.2 Provide the composed shim wasm

The bridge reads `POSTGIS_SHIM_WASM` / `MOBILITYDB_SHIM_WASM` at
`.load` time to locate the composed shim. Two deployment options:

**Option A — co-locate.** Drop the composed wasm next to the
bridge and point the env var at it:

```sh
cp ~/git/datafission/extensions/postgis/deps/postgis-composed.wasm \
   ~/git/sqlink/extensions/postgis-bridge/
cp ~/git/datafission/extensions/mobilitydb/deps/mdb-temporal-wasm.wasm \
   ~/git/sqlink/extensions/mobilitydb-bridge/
```

**Option B — system path.** Put the composed wasm under a shared
prefix (e.g. `/usr/local/share/sqlink/`) and let the env var point
there. Lighter per-install footprint when both bridges share the
postgis composed wasm (mobilitydb's bridge transitively references
postgis).

### 4.3 Load from SQLite

SQLite has no signing requirement, so loading is straightforward:

```sh
POSTGIS_SHIM_WASM=~/git/sqlink/extensions/postgis-bridge/postgis-composed.wasm \
    /opt/homebrew/bin/sqlite3 -cmd \
    ".load ~/git/sqlink/extensions/postgis-bridge/libpostgis_sqlite_bridge.dylib" \
    :memory:

sqlite> SELECT ST_AsText(ST_GeomFromText('POINT(1 2)'));
POINT(1 2)
```

For mobilitydb, both env vars need to be set since the mobilitydb
shim transitively uses postgis types:

```sh
POSTGIS_SHIM_WASM=~/git/sqlink/extensions/postgis-bridge/postgis-composed.wasm \
MOBILITYDB_SHIM_WASM=~/git/sqlink/extensions/mobilitydb-bridge/mobilitydb-composed.wasm \
    /opt/homebrew/bin/sqlite3 :memory:

sqlite> .load ~/git/sqlink/extensions/postgis-bridge/libpostgis_sqlite_bridge.dylib
sqlite> .load ~/git/sqlink/extensions/mobilitydb-bridge/libmobilitydb_sqlite_bridge.dylib
sqlite> SELECT tfloat_max_agg(v) FROM (VALUES (1.0), (3.0)) AS t(v);
```

The order of `.load` calls matters — load `postgis-bridge` before
`mobilitydb-bridge` so the GEOMETRY type is registered before
mobilitydb's tgeompoint conversions reference it.

## 5. The SQL preprocessor (optional but recommended)

SQLite doesn't speak PostgreSQL's `&&` overlap operator or
`CAST(text AS GEOMETRY)` shorthand. The shim's WIT surface speaks
function calls only. `shim-sql-preprocess` is a thin CLI that
rewrites these at the SQL-text level before SQLite sees the query:

```sh
cd ~/git/shim-sql-preprocess
cargo build --release

# Then pipe SQL through it:
echo "SELECT a && b FROM t" | \
    ./target/release/shim-sql-preprocess \
    --interface /tmp/postgis-interface.sqlite \
    --dialect sqlite
# → SELECT st_bboxintersects(a, b) FROM t
```

For interactive use the preprocessor is optional — users who write
the function-call form directly don't need it. For drop-in
compatibility with PostGIS query workloads, gate every SQL through
it. The smoke runner does this when `SHIM_SQL_PREPROCESS` is set.

## 6. Regeneration workflow

When the upstream shim's SQL surface changes (new functions, new
types, signature updates):

```sh
# 1. Rebuild the upstream shim composed wasm if upstream moved
cd ~/git/datafission
scripts/rebuild-postgis-composed.sh           # postgis
extensions/mobilitydb/scripts/bump-pin.sh     # mobilitydb

# 2. Rebuild the interface DB (the shim's published surface)
#    — published from the shim itself or DataFission's extension
#    catalog dump.

# 3. Regenerate both SQLite bridges
cd ~/git/sqlink-shim-codegen
cargo run --release -- --interface /tmp/postgis-interface.sqlite \
                       --out ~/git/postgis-sqlite-bridge
cargo run --release -- --interface /tmp/mobilitydb-interface.sqlite \
                       --out ~/git/mobilitydb-sqlite-bridge

# 4. Rebuild + smoke (see next section)
```

Each regen produces an idempotent crate (rustfmt-clean, same input
→ same output). Diff against the prior bridge to see what upstream
changed.

## 7. Smoke testing

Before redeploying into sqlink, run the smoke runner:

```sh
cd ~/git/shim-bridge-smoke-tests
make postgis-sqlite mobilitydb-sqlite
```

Output should end with `===== ALL SMOKE TESTS PASSED =====`.

The Makefile takes overrides via env var when artifacts live in
non-default locations:

```sh
POSTGIS_SQLITE_BRIDGE=/custom/path/libpostgis.dylib \
POSTGIS_SHIM=/custom/path/postgis-composed.wasm \
    make postgis-sqlite
```

The smoke cases live under `cases/<shim>/`. Adding a new case is
just dropping in `<n>.sql` + `<n>.expected`. The runner diffs
actual vs expected and reports per-case PASS/FAIL.

The shim-sql-preprocess CLI hooks in automatically when both the
`SHIM_SQL_PREPROCESS` and `SHIM_INTERFACE_DB` env vars are set;
the Makefile threads them through by default. Mark a case
`<n>.no-preprocess` (empty marker file) to skip preprocessing for
cases that should reach the bridge as-typed.

SQLite-only cases (e.g. UDTFs which sqlink ships but ducklink
scaffolds) live under `cases/postgis-sqlite-only/` and run from
`make postgis-sqlite` automatically.

## 8. Troubleshooting

### "unknown import: env::memory_budget"

The bridge's wasmtime instance can't find a guest env import. Means
the composed wasm doesn't match what the bridge was generated
against. Action: confirm both came from the same upstream pin
(check the bridge's `Cargo.toml` git revision against the composed
wasm's `*-pin.txt` file).

### "out of bounds memory access" on `.load`

The composed shim's memory min size exceeds what wasmtime got. This
is almost always a pool-sizing mismatch — the composed shim wants
~21 MiB and the bridge defaulted to less. Regen and rebuild.

### macOS `/usr/bin/sqlite3` refuses `.load`

System sqlite is built with `-DSQLITE_OMIT_LOAD_EXTENSION`. Use
homebrew sqlite at `/opt/homebrew/bin/sqlite3`.

### "type_id mismatch" on a typed input

The bridge was generated against an older interface DB; signatures
have moved. Regen the bridge from the current interface DB.

### Both PostGIS + MobilityDB loaded but `tgeompoint` round-trip drops SRID

Make sure `postgis-bridge` loaded **before** `mobilitydb-bridge`.
The mobilitydb shim references the postgis GEOMETRY type
registration; loading them out of order leaves mobilitydb without
the type and SRID-bearing values come back as plain BLOB.

### `make smoke` reports `PASS: 0 / FAIL: 0`

The cases dir was empty (no `*.sql` files matched the case pattern).
Check the path passed to the runner.

## 9. Versioning

| Component | Version source |
|---|---|
| Composed shim wasm | Pin file in `extensions/<shim>/deps/<shim>-*-pin.txt` (upstream_commit + blake3) |
| Bridge crate | `Cargo.toml::package.version` — bumped by codegen when the interface DB shape changes |
| Interface DB | Shim-side; consult the shim repo's release tooling |

For a release commit, bump the pin file, regen both bridges,
rebuild, run smoke, and ship the two bridge artifacts + the
composed shim alongside this guide.
